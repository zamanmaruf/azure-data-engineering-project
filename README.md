Spotify End‑to‑End Azure Data Engineering Platform (Databricks + CI/CD)


This repository contains an end‑to‑end data engineering platform built on Azure using Spotify data as the source. The goal of the project is to go beyond a simple ETL and model what a production‑style analytics platform looks like: raw data ingestion, medallion‑layered transformations, governed access, and automated CI/CD for Databricks assets. The pipeline lands raw Spotify data in Azure, processes it through bronze, silver, and gold layers in Azure Databricks, and publishes curated tables into Azure SQL Database for downstream analytics and reporting.

At the heart of the project is Azure Databricks, which serves as the main transformation engine. Unity Catalog is used to manage catalogs, schemas, and access control, while Delta Lake and Delta Live Tables provide reliable, incremental processing across the medallion layers. Bronze tables capture raw ingested data, silver tables standardize and clean that data, and gold tables expose business‑friendly views such as top tracks, artists, and listening patterns. For scenarios that require fresh insights, the solution also includes streaming components built with Spark Structured Streaming and Delta to move toward near real‑time analytics.

Ingestion and orchestration are handled by Azure Data Factory, which schedules and coordinates data movement into the lakehouse and triggers the Databricks workloads. Azure SQL Database acts as the serving layer, hosting denormalized and aggregated tables that are optimized for BI tools and ad‑hoc queries. This separation between the transformation lakehouse (Databricks + Delta) and the serving store (Azure SQL DB) mirrors how many production architectures are designed, with one layer focused on scalable processing and another focused on low‑latency consumption and reporting.

A key design choice in this project is to treat the entire data platform as code. All Databricks resources—jobs, pipelines, and related assets—are defined using Databricks Asset Bundles. The bundle configuration lives in this repository and describes how those resources should be deployed into different environments. Instead of manually configuring jobs in the Databricks UI, the project uses declarative YAML to specify resources, paths, catalogs, schemas, and environment‑specific settings, making the platform easier to version, review, and repeat.

Continuous integration and deployment are implemented with GitHub Actions. A dedicated workflow validates the Databricks Asset Bundle on pull requests and pushes to the main branch, and on successful pushes it deploys the bundle to a dev target and then to a prod target. The workflow uses the Databricks CLI and the official setup action to connect to the Azure Databricks workspace, authenticate with a token, and run databricks bundle validate and databricks bundle deploy against the appropriate target configuration. This approach follows Databricks’ recommended pattern for CI/CD and ensures that changes are validated and deployed through GitHub instead of ad‑hoc UI changes.

As part of getting this CI/CD flow working, the project also tackles some of the practical challenges that come up in real environments. For example, when customizing bundle deployment paths, the workflow initially failed with permission errors when trying to upload bundle files into the .bundle directory under the workspace. Resolving this required understanding development vs. production deployment modes, ensuring dev targets use user‑unique root paths, and aligning workspace permissions and bundle permissions with the identity used by GitHub Actions. Those permission and ACL adjustments are now reflected in the bundle configuration and deployment strategy captured in this repo.

From an educational and portfolio perspective, this project is intended to show how multiple Azure services and Databricks capabilities fit together into a single, coherent platform. It demonstrates cloud‑native ingestion, scalable Spark and PySpark transformations, medallion architecture, Unity Catalog governance, a BI‑ready serving layer on Azure SQL Database, and CI/CD with Databricks Asset Bundles and GitHub Actions. The repository is structured so that you can explore the orchestration pieces, the Databricks notebooks and pipelines, and the bundle configuration that defines and deploys the platform end to end. It is a work in progress, but it already reflects many of the best practices used by modern data engineering teams building production‑style systems on Azure and Databricks.
